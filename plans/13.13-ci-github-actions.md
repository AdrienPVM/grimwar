# Plan 13.13 — CI GitHub Actions (câblage des 3 cadences livrées par 13.12)

> **Sprint** : S1 (outillage / qualité). **Statut** : en cours.
>
> **Pourquoi maintenant** : la matrice combinatoire L1 livrée par 13.12 (85 invariants sur 19 personas, garde-fou anti-dérive « axe matrice ≡ axe bundle ») est un **garde-fou en sursis** tant qu'elle ne s'exécute pas automatiquement. La première modification qui ajoute une 13ᵉ classe au bundle sans persona passera sans rien casser tant que personne ne lance `pnpm test:matrix` localement. Toute la valeur de 13.12 est conditionnée à ce plan. C'est le seul candidat qui **protège le travail déjà poussé** plutôt que d'en ajouter.

## Goal

Câbler en GitHub Actions les 3 cadences npm livrées par 13.12 + les gates de sécurité (rules) et e2e, sur un **flow mixte** : direct-push autorisé par défaut, branch protection bloquante sur les paths critiques (`public/data/**`, `scripts/data/srd-*.ts`, `.github/workflows/**`).

Livrable : un workflow `.github/workflows/ci.yml` à **4 jobs parallèles bloquants** (`static` / `unit` / `matrix` / `emulator`), validé par un **premier run réel sur PR draft** (un workflow ne se prouve pas vert par la gate locale — H1).

## Verdict d'inventaire (vérification des prémisses du cadrage 13.9 sur disque)

Discipline identique à 13.10/13.12 : chaque prémisse du cadrage oral (notes 13.9 + `tests/README.md`) confrontée au repo réel avant toute ligne. **Aucun stub `plans/13.13-*.md` n'existait** — 13.13 n'était cadré que dans les `## Notes` de 13.12 et les cross-refs de 13.14. Prémisses fausses/périmées :

- **P3 (split de cadence) — IMPRÉCISE puis tranchée nécessité, pas préférence.** `pnpm test` complet **n'est pas exécutable tel quel en CI** : il embarque `tests/firestore-rules.test.ts`, qui se **skippe silencieusement** sans `FIRESTORE_EMULATOR_HOST` (`firestore-rules.test.ts:37-44`) → couverture rules perdue sans bruit. La partition `fast ∪ matrix ∪ rules` couvre exactement tous les tests Vitest, chacun **une seule fois**. Le split Q1 est donc *structurellement nécessaire*.
- **P4 (anti-flake) — LARGEMENT DÉJÀ FAIT.** `workers:1` (`playwright.config.ts:46`) ✅ et `retries: CI?1:0` (`playwright.config.ts:42`) ✅ s'activent seuls car GitHub Actions exporte `CI=true`. La « quarantaine après 3 fails » n'a aucun équivalent natif Playwright (pas de mémoire cross-run) → **politique manuelle** (`test.fixme` + issue), pas de l'outillage. Ce volet se réduit à : exporter les artefacts trace/report on-failure (la config produit déjà les traces) + documenter la politique de quarantaine. → fondu dans les commits 3 (artefact) et 4 (doc).
- **P5 (« zéro secret = pas besoin de credentials ») — PARTIELLEMENT FAUSSE.** `env.ts:28-33` jette sur **toute** var `VITE_FIREBASE_*` requise vide (6 requises). Le `webServer` Playwright n'injecte que 2 vars (`PROJECT_ID`, `USE_FIREBASE_EMULATOR`) ; les autres viennent de `.env.local`, **absent en CI** → `pnpm dev` (serveur e2e) crashe au boot. Correctif : fournir **5 valeurs factices non-secrètes** en env clair du workflow (pas des GitHub Secrets). Option A tient (zéro secret réel), mais « pas besoin de credentials » sous-estime : il faut des placeholders de *présence*.
- **P6 (firebase-tools dispo) — FAUSSE (dépendance non déclarée).** `firebase-tools` n'est **pas** dans `package.json` (global sur le Mac d'Adrien). CI doit l'installer explicitement **+ setup Java** (`actions/setup-java`, l'émulateur en a besoin). Bonne nouvelle : `emulators:exec` **attend déjà** que l'émulateur soit prêt avant de lancer la commande — pas de healthcheck/sleep manuel à écrire, le piège « émulateur lent » est résolu structurellement par le script existant.
- **P7 (conflits de ports) — RISQUE FAIBLE.** Runner neuf = pas de conflit. `--only firestore,auth` ne démarre ni functions(5001) ni UI(4000). `--strictPort` sur 5179. Non-sujet sur un runner propre.
- **Constat Node** : `engines.node: ">=22 <26"`. La machine par défaut d'Adrien (v26) est **hors range supporté** → `setup-node@22` est le seul Node conforme, pas un confort. Gate locale par `nvm exec 22`.

## Décisions de cadrage (validées avec Adrien avant exécution)

- **Q1 — `test:fast` + `test:matrix` SÉPARÉS** sur chaque déclencheur (pas `pnpm test` complet). Signal fin (fast rouge = algo unitaire ; matrix rouge = couverture combinatoire) + parallélisme + cohérence avec les 3 cadences gravées en 13.12. Renforcé par la nécessité structurelle ci-dessus (P3).
- **Q2 — `test:rules` tourne en CI.** Les rules Firestore sont la sécurité du modèle de données ; une régression qui passe = trou de sécurité potentiel en prod. Coût marginal nul (l'émulateur tourne déjà pour les e2e). Satisfait trivialement la discipline « test:rules avant deploy rules ».
- **Arbre des jobs (plomberie tranchée par GSD)** : un seul workflow `ci.yml`, **4 jobs parallèles, tous bloquants**, triggers `push: main` + `pull_request`, `concurrency` + `cancel-in-progress`.

  | Job | Contenu | Émulateur/Java | Coût |
  |---|---|---|---|
  | `static` | typecheck + lint | non | léger |
  | `unit` | `test:fast` | non | léger |
  | `matrix` | `test:matrix` ← *le garde-fou de 13.12* | non | moyen |
  | `emulator` | `test:rules` puis `test:e2e` | oui (Java + firebase-tools + chromium) | lourd |

  - `unit` et `matrix` séparés → honore Q1 (signal fin + parallélisme).
  - `rules` + `e2e` groupés dans **un** job `emulator` → setup lourd (Java + CLI + browsers + jars) payé **une fois** ; la granularité Q1 est préservée par les statuts par-step de GitHub. `rules` tourne en premier (quelques s) puis `e2e`.
  - `static` isolé → typecheck/lint ne traînent jamais Java, feedback ~30 s.
  - Caching par job : pnpm store keyed sur `pnpm-lock.yaml` ; `emulator` ajoute `~/.cache/ms-playwright` (chromium) + `~/.cache/firebase/emulators` (jars).

- **H1 — branche `ci/13.13` + PR draft.** Un workflow ne se prouve PAS vert par la gate locale (ajouter un `.yml` ne touche ni typecheck, ni test, ni lint, ni e2e → gate verte trivialement, ne prouve rien sur le workflow). Le vrai « rouge avant vert » d'un workflow = son premier run réel sur GitHub. On observe les 4 jobs sur PR draft avant tout merge sur `main`. Version CI exacte du dispositif `uat-review/`.
- **H2 — « bloquant » est un réglage de repo, pas du fichier.** Le `ci.yml` *définit* les jobs ; les rendre requis pour merger = activer « required status checks » dans la branch protection GitHub (UI/API, action sortante d'Adrien). Liste des 4 checks à cocher fournie une fois le workflow vert : `static`, `unit`, `matrix`, `emulator`.
- **H3 — flow MIXTE.** Direct-push sur `main` autorisé par défaut. Paths protégés exigeant PR : `public/data/**`, `scripts/data/srd-*.ts`, `.github/workflows/**`. Justification : ce sont les bundles SRD durcis par 13.7-13.10b + la pipeline CI elle-même — une régression silencieuse y coûte cher et la CI seule en post-hoc serait trop tard. Pour les autres paths, le coût d'une PR systématique en solo dépasse le bénéfice. **À acter dans CLAUDE.md (nouvelle politique projet).**
