# Plan 13.7 — Schema character + scripts d'extraction SRD reproductibles

> **Plan A du jalon 1 « complétude SRD 5.2.1 niveau 1 ».** Premier des 5 plans (13.7→13.11) insérés en suffixe S1 suite à `docs/AUDIT-SRD-COMPLETUDE.md`. Bloque 13.8, 13.9, 13.10, 13.11 — tous consomment les types + scripts livrés ici.

## Goal

Étendre le `Character` schema pour porter **tous** les sous-choix imposés par le SRD 5.2.1 au niveau 1 (ascendances + classes), bumper `schemaVersion` (1 → 2), aligner `firestore.rules > characterShapeOK`, livrer la migration Dexie + Firestore, **et** poser les scripts d'extraction SRD reproductibles (`scripts/extract-srd-*.ts`) qui régénèrent `public/data/*.json` enrichis depuis les deux PDF SRD seuls. Aucun changement UI à ce plan — 13.7 est de la pure infrastructure data + types pour que 13.8 et 13.9 aient un terrain prêt.

**Critère d'utilité** : à la fin du plan, `grep -r 'subancestryId' src/` doit confirmer que le champ a disparu du `Character` schema au profit de sous-choix structurés par trait, et `pnpm tsx scripts/extract-srd-ancestries.ts` doit régénérer `public/data/ancestries.json` avec un tableau `options[]` machine-readable sur chaque trait à sous-choix (Drakéide → `dragonAncestries[]` de 10 entrées, etc.) à partir des seuls `content-sources/pdfs/SRD_CC_v5.2.1.pdf` + `FR_SRD_CC_v5.2.1.pdf`.

### §0.1 — Liste exhaustive et définitive des champs schema (FIGÉE)

**Décisions de structure (Adrien 2026-05-17)** :
- **Sous-objets groupés**, pas champs à plat. Création de `ancestrySubChoices: {...}` et de sous-choix de classe portés par chaque entrée du tableau `classes[]`. Raisons : racine du personnage lisible, migration en un bloc, Zod `discriminatedUnion` natif sur `ancestryId` (resp. `classId`).
- **Sous-choix de classe portés par `classes[]`**, pas par la racine. `weaponMasteries`, `expertiseSkills`, `fighterFightingStyle`, `clericDivineOrder`, `druidPrimalOrder`, `eldritchInvocations`, `wizardSpellbookL1` vivent dans chaque entrée du tableau `classes[]`. Raison : sémantiquement correct en multi-classe — un Roublard 2 / Magicien 1 a son expertise et ses 2 weapon masteries du côté Roublard, zéro côté Magicien. Le level-up (plan 18) en aura besoin.
- **`extraLanguages` reste racine** : agrège plusieurs sources (Roublard, Background, Origin Feat Skilled, etc.) — l'attribution par classe n'a pas de sens utile.
- **`ancestryCastingAbility` / `ancestryExtraSkill` / `ancestrySize` restent dans `ancestrySubChoices`** : ils dérivent de l'ascendance, pas d'une classe.
- **Retrait** : `subancestryId: slug.nullable()` (SRD ne modélise pas les lignages comme sous-ascendances ; cf. audit § E.5).
- **`schemaVersion`** : bump `z.literal(1)` → `z.literal(2)`.

**Structure finale figée** :

```ts
// Racine Character v2 (extrait des champs nouveaux/modifiés uniquement)
{
  schemaVersion: z.literal(2),
  ancestryId: slug,
  // subancestryId: RETIRÉ
  ancestrySubChoices: AncestrySubChoicesSchema, // sous-objet typé, discriminated union sur ancestryId
  extraLanguages: z.array(slug), // racine — agrège toutes les sources
  classes: z.array(CharacterClassEntrySchema), // chaque entrée porte ses sous-choix
  // ... reste inchangé
}
```

**`AncestrySubChoicesSchema`** — discriminated union sur `ancestryId` :
| `ancestryId` | Champs requis | Type / valeurs admises |
|---|---|---|
| `dragonborn` | `dragonAncestry` | enum 10 : `black\|blue\|brass\|bronze\|copper\|gold\|green\|red\|silver\|white` |
| `tiefling` | `tieflingLegacy`, `ancestryCastingAbility`, `ancestrySize` | enum 3 : `abyssal\|chthonic\|infernal` ; enum 3 : `int\|sag\|cha` ; enum 2 : `small\|medium` |
| `elf` | `elfLineage`, `ancestryCastingAbility`, `ancestryExtraSkill` | enum 3 : `drow\|high-elf\|wood-elf` ; enum 3 ; slug ∈ {`insight`,`perception`,`survival`} |
| `gnome` | `gnomeLineage`, `ancestryCastingAbility` | enum 2 : `forest\|rock` ; enum 3 |
| `goliath` | `goliathAncestry` | enum 6 : `cloud\|fire\|frost\|hill\|stone\|storm` |
| `human` | `ancestryExtraSkill`, `ancestrySize` | slug (1 parmi 18 skills SRD) ; enum 2 |
| `dwarf`, `halfling`, `orc` (et autres) | — | objet vide `{}` (pas de sous-choix L1 imposé) |

Source SRD : `docs/AUDIT-SRD-COMPLETUDE.md` § A.2.

**`CharacterClassEntrySchema`** — extension de l'entrée actuelle de `classes[]`, ajout des champs suivants (tous présents systématiquement, valeurs sentinelles si non-applicable à la classe) :

| Champ | Type | Valeurs admises | Requis pour | Sentinelle |
|---|---|---|---|---|
| `clericDivineOrder` | `enum \| null` | `protector\|thaumaturge` | `classId === 'cleric'` | `null` |
| `druidPrimalOrder` | `enum \| null` | `magician\|warden` | `classId === 'druid'` | `null` |
| `fighterFightingStyle` | `slug \| null` | feat id ∈ `archery\|defense\|great-weapon-fighting\|two-weapon-fighting` | `classId === 'fighter'` | `null` |
| `weaponMasteries` | `slug[]` | weapon ids (37 armes éligibles SRD) | `classId ∈ {barbarian,fighter,paladin,ranger,rogue}` (count selon classe) | `[]` |
| `expertiseSkills` | `slug[]` | skill ids déjà maîtrisés par le perso, taille 2 | `classId === 'rogue'` | `[]` |
| `eldritchInvocations` | `slug[]` | invocation ids, taille 1 | `classId === 'warlock'` | `[]` |
| `wizardSpellbookL1` | `slug[]` | spell ids L1, taille 6 | `classId === 'wizard'` | `[]` |

Source SRD : `docs/AUDIT-SRD-COMPLETUDE.md` § B.2, C.1, C.2, C.3.

**Counts de Weapon Mastery par classe à L1** (figés depuis SRD § C.1) : Barbare 2, Guerrier 3, Paladin 2, Rôdeur 2, Roublard 2. Monk **n'a pas** Weapon Mastery → sentinelle `[]`.

### §0.2 — Stratégie de migration Dexie + Firestore (FIGÉE — Adrien 2026-05-17)

**Allègement validé** : on **NE construit PAS** de step de rattrapage dans le wizard. Lyralei + persos de test UAT seront **supprimés et recréés à neuf**. Aucune fiche utilisateur réel n'est en base, l'app n'est pas publiée. Construire un mécanisme de migration UI pour 3 fiches jetables = sur-effort rejeté.

**Ce qui est GARDÉ** (non-négociable) :
1. **Migration `schemaVersion: 1 → 2` lazy à l'ouverture** :
   - Dexie : bump de version de la DB locale + hook `upgrade` qui injecte les sentinelles sur les fiches `schemaVersion: 1` en cache. Idempotent.
   - Firestore : dans `src/features/sheet/use-character.ts`, à la lecture, détecter `schemaVersion === 1` → cloner en mémoire avec sentinelles (`ancestrySubChoices: {}`, `extraLanguages: []`, et pour chaque entrée `classes[]` : sous-choix sentinelles) → écrire immédiatement v2 (one-shot, idempotent). Log via `event-logger.ts` (`kind: 'schema.upgraded'`).
   - Plus aucun `setDoc` ne pousse `schemaVersion: 1`.
2. **Tolérance des sentinelles par TOUS les modes de la fiche** (Combat / Magie / Essence / Avoir / Âme) : un mode qui itère `classes[0].weaponMasteries` voit `[]` → n'affiche rien, jamais une erreur. Robustesse de base, non négociable. Couvert par tests unit sur chaque mode.
3. **Durcissement de `wizard-validation.ts`** : le wizard **refuse** de submit un perso dont un sous-choix SRD requis est en sentinelle. Garantie : on ne crée jamais de perso SRD-incomplet. Implémenté à 13.8/13.9 mais consigné ici.

**Ce qui est RETIRÉ** : pas de step de rattrapage conditionnel injecté dans le wizard pour compléter une vieille fiche. Un perso v1 s'ouvre proprement (sentinelles vides) ; pour le compléter, on le recrée à neuf, on ne le « répare » pas dans le wizard. Plans 13.8 et 13.9 mis à jour en cohérence.

**Conséquences pour le code** :
- `src/shared/lib/dexie-db.ts` : bump `version(N)` → `version(N+1)`, hook `upgrade` strict (test unit en rouge avant fix).
- `src/features/sheet/use-character.ts` : branche `if (schemaVersion === 1) → upgradeToV2(doc) → setDoc(v2) → return v2`.
- `src/features/wizard/submit-from-wizard.ts` : payload toujours `schemaVersion: 2` avec sentinelles bien formées.
- **Aucune Cloud Function batch.** Plan 14 sera le premier à exiger une migration côté Cloud Function ; pas ici.
- **Aucun nouvel index Firestore** requis (les nouveaux champs ne sont pas filtrés/triés). Vérification dans `firestore.indexes.json`.
