# Plan 13.7 — Schema character + scripts d'extraction SRD reproductibles

> **Plan A du jalon 1 « complétude SRD 5.2.1 niveau 1 ».** Premier des 5 plans (13.7→13.11) insérés en suffixe S1 suite à `docs/AUDIT-SRD-COMPLETUDE.md`. Bloque 13.8, 13.9, 13.10, 13.11 — tous consomment les types + scripts livrés ici.

## Goal

Étendre le `Character` schema pour porter **tous** les sous-choix imposés par le SRD 5.2.1 au niveau 1 (ascendances + classes), bumper `schemaVersion` (1 → 2), aligner `firestore.rules > characterShapeOK`, livrer la migration Dexie + Firestore, **et** poser les scripts d'extraction SRD reproductibles (`scripts/extract-srd-*.ts`) qui régénèrent `public/data/*.json` enrichis depuis les deux PDF SRD seuls. Aucun changement UI à ce plan — 13.7 est de la pure infrastructure data + types pour que 13.8 et 13.9 aient un terrain prêt.

**Critère d'utilité** : à la fin du plan, `grep -r 'subancestryId' src/` doit confirmer que le champ a disparu du `Character` schema au profit de sous-choix structurés par trait, et `pnpm tsx scripts/extract-srd-ancestries.ts` doit régénérer `public/data/ancestries.json` avec un tableau `options[]` machine-readable sur chaque trait à sous-choix (Drakéide → `dragonAncestries[]` de 10 entrées, etc.) à partir des seuls `content-sources/pdfs/SRD_CC_v5.2.1.pdf` + `FR_SRD_CC_v5.2.1.pdf`.

## §0 Préambule schéma — POINT D'ARRÊT (aucun code avant validation Adrien)

> **Ce préambule N'EST PAS un step `[ ]`.** C'est un livrable rédigé. Le plan code ne démarre qu'après validation explicite d'Adrien sur cette section. Toute modification doit faire l'objet d'un échange avant exécution.

**Forme du livrable §0** : un document interne (rédigé dans ce plan, sous-section §0.1 à §0.4) couvrant :

### §0.1 — Liste exhaustive et définitive des champs schema à ajouter / modifier / retirer

Pour chaque champ : **nom**, **type TypeScript / Zod**, **requis ou optionnel**, **valeurs admises** (énumération ou contrainte), **entité concernée** (Character racine, sous-objet `ancestrySubChoices`, sous-objet `classSubChoices`, etc.), **source SRD** (référence à `docs/AUDIT-SRD-COMPLETUDE.md` section X.Y).

Champs à proposer **(liste de travail — Adrien fige la liste finale en validation §0)** :

- **Retrait** : `subancestryId: slug.nullable()` (devient inutile — SRD ne modélise pas les lignages comme sous-ascendances ; cf. audit § E.5).
- **Sous-choix d'ascendance** (un sous-objet `ancestrySubChoices` typé discriminant sur `ancestryId`, ou champs racine — choix à arbitrer en §0) :
  - `dragonAncestry`: enum 10 valeurs (`black|blue|brass|bronze|copper|gold|green|red|silver|white`) — requis si `ancestryId === 'dragonborn'`.
  - `tieflingLegacy`: enum 3 valeurs (`abyssal|chthonic|infernal`) — requis si `ancestryId === 'tiefling'`.
  - `elfLineage`: enum 3 valeurs (`drow|high-elf|wood-elf`) — requis si `ancestryId === 'elf'`.
  - `gnomeLineage`: enum 2 valeurs (`forest|rock`) — requis si `ancestryId === 'gnome'`.
  - `goliathAncestry`: enum 6 valeurs (`cloud|fire|frost|hill|stone|storm`) — requis si `ancestryId === 'goliath'`.
  - `ancestryCastingAbility`: enum 3 valeurs (`int|sag|cha`) — requis si ascendance pose une option d'incantation (Tieffelin, Elfe, Gnome).
  - `ancestryExtraSkill`: slug (skill id) — requis si Elfe (Keen Senses) ou Humain (Skillful) ; nullable sinon.
  - `ancestrySize`: enum 2 valeurs (`small|medium`) — requis si ascendance à taille variable (Humain, Tieffelin) ; nullable sinon.
- **Sous-choix de classe** :
  - `clericDivineOrder`: enum 2 valeurs (`protector|thaumaturge`) — requis si `classes[].classId === 'cleric'`.
  - `druidPrimalOrder`: enum 2 valeurs (`magician|warden`) — requis si `classes[].classId === 'druid'`.
  - `fighterFightingStyle`: slug (feat id parmi 4 SRD : `archery|defense|great-weapon-fighting|two-weapon-fighting`) — requis si `classes[].classId === 'fighter'`.
  - `weaponMasteries`: tableau de slug (weapon ids), taille variable selon classe (Barbare 2, Guerrier 3, Paladin 2, Rôdeur 2, Roublard 2) — requis quand la classe en reçoit à L1 ; tableau vide sinon.
  - `expertiseSkills`: tableau de 2 slugs (skill ids) — requis si Roublard ; tableau vide sinon.
  - `eldritchInvocations`: tableau de 1 slug (invocation id parmi 5 éligibles L1) — requis si Warlock ; tableau vide sinon.
  - `extraLanguages`: tableau de slugs (language ids), taille variable — alimenté par Roublard (+1) et toute autre source (Background, Origin Feat Skilled, etc.).
  - `wizardSpellbookL1`: tableau de slugs (spell ids), taille 6 — requis si Magicien L1 ; sinon vide. Distinct de `knownSpells['wizard']` (qui contient les 4 préparés).
- **`schemaVersion`** : bump `z.literal(1)` → `z.literal(2)`.

§0.1 doit aussi statuer sur :
- **Sous-objet typé vs champs racine** : un objet `ancestrySubChoices: { dragonAncestry?, tieflingLegacy?, ... }` est-il préféré à 8 champs racine ? Arguments à peser : lisibilité fiche (groupé), discrimination par `ancestryId` (Zod `discriminatedUnion`), migration plus simple en un seul objet, vs nuance « le sous-choix appartient au perso, pas à l'ascendance ». Idem pour `classSubChoices` avec `classes[].classId`. Adrien tranche.
- **Position dans le multi-class** : `weaponMasteries`/`expertiseSkills`/`fighterFightingStyle` sont-ils racine (`character.weaponMasteries`) ou portés par chaque entrée du tableau `classes[]` (`classes[0].weaponMasteries`) ? La 2e option est plus correcte sémantiquement (un Roublard 2 / Magicien 1 a 2 weapon masteries du Roublard et 0 du Magicien). Adrien tranche.

### §0.2 — Stratégie de migration Dexie + Firestore

- **Dexie** : `src/shared/lib/dexie-db.ts` — bump de version. Hook de migration : pour chaque doc dans `characters`, ajouter les nouveaux champs avec valeurs par défaut (`null`, `[]`, etc.) sans casser les fiches existantes locales. Aucune fiche n'a de sous-choix avant 13.8 → la valeur est sentinelle ; 13.8 forcera la complétion via le wizard avant que la fiche soit jouable.
- **Firestore (côté code applicatif)** :
  - Toute fiche `schemaVersion: 1` lue depuis Firestore est migrée à la volée côté `useCharacter` (one-shot upgrade, écriture immédiate avec `schemaVersion: 2` + sentinelles) — Adrien n'a pas de fiches en prod aujourd'hui, mais on garde la trace pour 0 régression.
  - Plus aucun `setDoc` ne pousse `schemaVersion: 1`.
- **Pas de Cloud Function de migration batch** — il n'y a pas de fiche multi-utilisateur à toucher en S1. Plan 14 (Campaigns model) sera le premier à exiger une migration côté Cloud Function ; pas ici.
- **Index Firestore** : aucun nouvel index n'est requis (les nouveaux champs ne sont pas filtrés/triés par query). À confirmer en §0.2 contre `firestore.indexes.json`.

### §0.3 — Impact sur `firestore.rules > characterShapeOK`

- Les nouveaux champs ajoutés au schéma doivent-ils être :
  - **(a)** ajoutés à `hasAll([...])` (donc requis) — alors une fiche sans Drakéide qui n'a pas `dragonAncestry` doit toujours présenter la clé (nullable accepté) ;
  - **(b)** simplement tolérés (rules silencieuses, validation côté Zod) — moins strict mais plus simple à déployer (pas de risque que la prod refuse une fiche L1 ancienne).
- Recommandation Claude Code (à valider en §0) : **(b)** — la rule garde sa simplicité (forme structurelle minimale), Zod côté client porte la validation fine, et tout sous-choix requis par le SRD est imposé par le wizard 13.8/13.9 avant qu'un perso atterrisse en Firestore. C'est cohérent avec la philosophie actuelle de `characterShapeOK` (commentaire ligne 116-122 du firestore.rules).
- **Conséquence déploiement (cf. D5)** : si on retient (b), aucun deploy rules n'est urgent à 13.7. Si Adrien tranche (a), il faut déployer les rules **avant** que 13.8 atterrisse (sinon les fiches L1 sans Drakéide refusées en prod). Le déploiement de rules doit se faire avec `pnpm test:rules` vert + `firebase deploy --only firestore:rules`.
- `tests/firestore-rules.test.ts` doit être étendu pour couvrir : ancien schéma `schemaVersion: 1` accepté (compat) ou refusé (durcissement) — selon arbitrage (b)/(a).

### §0.4 — Liste exhaustive des scripts d'extraction à livrer

- `scripts/extract-srd-ancestries.ts` — lit les 2 PDF SRD, sort `public/data/ancestries.json` enrichi avec `options[]` par trait à sous-choix (Drakéide → `dragonAncestries`, Tieffelin → `tieflingLegacies` avec sorts d'héritage par niveau, Elfe → `elfLineages` avec cantrip + sorts L3/L5, Gnome → `gnomeLineages`, Goliath → `giantAncestries`, Humain → `humanOptions`).
- `scripts/extract-srd-classes.ts` — sort `public/data/classes.json` enrichi avec `divineOrders[]`, `primalOrders[]`, sur les classes concernées + `weaponMasteryCount` (par classe à L1).
- `scripts/extract-srd-feats.ts` — sort `public/data/feats.json` enrichi à 17 entrées (4 Origin + 2 General + 4 Fighting Style + 7 Epic Boon), avec un champ `category` pour discriminer.
- `scripts/extract-srd-invocations.ts` — sort `public/data/invocations.json` (NOUVEAU) à 28 entrées avec `prerequisiteWarlockLevel: number | null` (5 entrées à `null` = éligibles L1).
- `scripts/extract-srd-weapon-mastery.ts` — sort `public/data/items.json` (mise à jour des entrées arme existantes) avec `masteryProperty: enum 8 valeurs` sur chacune des 37 armes éligibles.
- **Pas de script `extract-srd-spells.ts`** ici — c'est le périmètre du plan 13.10. 13.7 livre l'infrastructure (helpers PDF→JSON), 13.10 livre l'extraction spells qui s'appuie dessus.
- **Pattern commun** : entrée `pnpm tsx scripts/extract-srd-<entity>.ts`, sortie déterministe (snapshot stable, idempotent, sortie triée par id), parse strict (échoue si une entrée attendue est absente du PDF — pas de défaut silencieux), versionnement des maps EN/FR dans `scripts/maps/srd-<entity>-en-fr.ts` quand applicable.
