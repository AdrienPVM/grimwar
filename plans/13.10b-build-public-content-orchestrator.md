**🔴 BLOCKER ancestries (verdict catégorie 2).** Le commit 13.10/4 (`5fafbc9`) a édité
`public/data/ancestries.json` **directement** (`rayon-de-maladie`→`rayon-empoisonne`,

**Rectification (avant commit 2)** : remonter le fix dans `scripts/data/srd-ancestries-l1.ts`
(valeurs déjà vérifiées contre `FR_SRD_CC_v5.2.1.txt` en 13.10/4) + maj des commentaires
`DEBT D9`. Aucune autre valeur ne change.

**⚠️ Note classes/ancestries — base non régénérable from-scratch.** Les enrichisseurs
`extract-srd-{classes,ancestries}.ts` lisent `public/data` et n'ajoutent que des champs
dérivés (`options`, features). La base 16-mai de `content-sources/extracted/srd` est
**schéma-périmée** : un re-merge la fait Zod-fail et vide le bundle (mesuré : `classes`
3069 lignes perdues, `ancestries` 804). Donc **la base curée vit dans `public/data`** ;
il n'existe pas de régénérateur from-scratch. **L'orchestrateur = « ré-appliquer tous les
producteurs connus idempotemment », pas « rebuild from raw ».** Rafraîchir la base
`content-sources` contre le schéma courant = travail futur **hors scope D17**.

**⚠️ Note magic-items/monsters — tension politique.** D17 critère #2 dit « merge AideDD
conservé pour `magic-items`/`monsters` ». Mais la « Politique de contenu (LOCKED) » de
CLAUDE.md interdit à toute **étape de build** de lire `content-sources/aidedd/*`. Un
orchestrateur qui re-merge depuis AideDD ferait de `content:build` une étape lisant AideDD
→ conflit avec la décision lockée. **Recommandation : pass-through** (l'orchestrateur lit
le `public/data/{magic-items,monsters}.json` live et le préserve, ne re-merge PAS depuis
AideDD). « Conservé » = préservé tel quel, pas re-mergé. `content:build` reste SRD-only ;
le bundle `magic-items` (251, grandfathered) est préservé. **À confirmer par Adrien** avant
commit 2 (conflit architectural avec décision lockée → relève de son arbitrage).
