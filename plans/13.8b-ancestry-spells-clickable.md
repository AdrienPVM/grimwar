# Plan 13.8b — Sorts d'ascendance consultables (Option B)

> **Mini-plan de finition du plan 13.8** (rendu sheet des sous-choix d'ascendance). Pas un sous-commit de 13.9 — traçable à part, traité et clos avant de reprendre 13.9. Diagnostic 2026-05-18 : `AncestrySpellsCard` rend ses sorts en `<li>` statique non cliquable pour les 3 ascendances à sorts (Elfe / Gnome / Tieffelin). Cohérence sheet-render cassée : tout sort affiché par la fiche doit être consultable (modale détail).

## Goal

Tout sort qui apparaît sur la fiche d'un personnage est consultable d'un tap. Pour Elfe / Gnome / Tieffelin, **avec ou sans classe lanceuse** :

- La carte « Sorts d'ascendance » (carte de lignage) propose chaque sort en `<button>` qui ouvre `SpellDetailModal`.
- Les sorts d'ascendance apparaissent ALSO dans la `SpellList` générale du mode Magie, avec un chip de source visible (« Lignage Drow », « Héritage Infernal », etc.) distinct du chip « Préparé ».
- La `SpellList` se rend même pour un non-caster dès que `knownSpells.ancestry` est non vide (plus de placeholder « aucun art arcanique »).
- La modale détail s'ouvre. Le bouton « Lancer » est **désactivé** avec hint explicite (« Pas encore implémenté pour les sorts d'ascendance ») tant que la mécanique de slots / once-per-day / featureUsage pour les sorts L3/L5 du Tieffelin n'est pas livrée. Cette dette est tracée explicitement (cf. `plans/DEBT.md > D12`, owner = plan futur dédié).

**Règle invariante actée** : « tout sort connu est dans la liste générale, cliquable, avec sa source affichée ». La double présence (carte de lignage + liste générale) est intentionnelle ET lisible grâce au chip source : la carte = le « pourquoi », la liste = le « quoi ».

**Cas caster × ascendance (ajouté avant impl 2026-05-19)** — Quand la classe est lanceuse (ex. Magicien), le sort de lignage cohabite dans la `SpellList` avec les sorts de classe. Exigences :

- Le sort de lignage apparaît dans la liste générale avec son chip source distinct (« Lignage Drow » / « Héritage Infernal » / « Lignage Forêts »), **visuellement séparable** des sorts de classe (le chip source ascendance utilise un styling distinct de tout chip de classe).
- **Collision** : si le même `spellSlug` apparaît à la fois comme cantrip de classe ET comme sort d'ascendance (cas rare mais possible — un Magicien Elfe Drow qui choisit Lumières dansantes comme cantrip de classe), la liste affiche **une seule entrée** avec **les deux sources** lisibles (deux chips, ou un chip composite « Magicien + Lignage Drow »), pas deux lignes identiques sans distinction. Le détail (modale) reflète aussi les deux sources.

Ce cas est ce qui a induit Adrien en erreur à l'UAT 13.8 — un sort d'ascendance noyé dans la liste générale sans signal visuel = bug UX. Le test caster × ascendance ferme exactement ce trou.
