# Plan 13.11 — Système d'import de contenu custom (mécanisme générique, vide de contenu)

> **Plan E du jalon 1.** Livre **le mécanisme** d'import de contenu custom. Le dépôt ne bundle aucun contenu non-SRD (cf. politique de contenu, `CLAUDE.md` decision log). Ce plan installe le tuyau ; le contenu y passe à l'usage côté MJ via un pack JSON qu'il fournit.

## Goal

Permettre au MJ d'une campagne d'**importer un pack JSON de contenu custom** (sorts, classes, ascendances, dons, items, monstres, etc.) qui coexiste avec le SRD au moment de la création de personnage et du jeu. Le système est **vide de contenu** : aucun pack n'est livré dans le dépôt, aucun script de conversion non-SRD n'est livré dans ce plan, aucune référence à `content-sources/aidedd/*` n'apparaît dans le périmètre.

**Critère d'utilité** : un MJ ouvre `/campaigns/{id}/settings/content`, drop un fichier `grimwar-content-pack.json` valide, voit un aperçu (compteurs par catégorie), confirme l'import. Le wizard de création de personnage pour cette campagne propose alors les options custom **fusionnées** avec le SRD (un Drakéide à 10 sous-types SRD + 2 sous-types custom du pack apparaît comme un seul jeu unifié de 12 options). Un pack invalide est rejeté en entier avec un message d'erreur précis (catégorie / id / champ / raison).

**Hors périmètre 13.11** :
- Export de pack depuis une campagne (symétrie utile, notée en `## Notes for next plan` pour un plan futur).
- Tout script qui lit `content-sources/aidedd/*` ou tout PDF non-SRD (politique de contenu).
- Tout contenu pré-rempli livré dans le dépôt.
- Marketplace ou partage public de packs.

## Spécification du format `grimwar-content-pack.json`

> **Contrat verrouillé par Adrien (cf. consigne du 2026-05-17).** Ce qui suit est la source de vérité du validateur livré par ce plan. Aucune divergence permise sans validation explicite.

### Enveloppe du pack

```jsonc
{
  "formatVersion": 1,                          // entier, requis. Le validateur rejette une version inconnue.
  "meta": {                                    // requis. Informatif, affiché dans l'UI d'aperçu.
    "name": "Pack maison Adrien — campagne Côte des Épées",
    "author": "Adrien Paelinck",
    "description": "Contenu privé pour la table de mai",  // optionnel
    "createdAt": "2026-05-17T16:00:00Z"        // ISO 8601 requis
  },
  "content": {                                 // requis. Sous-objet par catégorie.
    "ancestries": [],
    "subancestries": [],
    "classes": [],
    "subclasses": [],
    "feats": [],
    "spells": [],
    "items": [],
    "magicItems": [],
    "monsters": [],
    "invocations": [],
    "fightingStyles": [],
    "backgrounds": [],
    "conditions": []
  }
}
```

Toute catégorie absente du sous-objet `content` est implicitement un tableau vide.

### Règles transverses à toute entité

1. **`id`** (string, requis) : kebab-case, **unique dans sa catégorie**. Le validateur rejette les doublons d'id intra-pack.
2. **`source`** (string, requis) : libre, identifie l'origine du contenu (ex. nom du pack ou abréviation). Sert à l'affichage et au filtrage. Jamais utilisé pour la légalité — la légalité est de la responsabilité de l'utilisateur qui fournit le pack.
3. **`name`** (objet i18n requis) : `{ fr: string, en: string }`. Chaque langue **non vide**. Le validateur rejette une string nue où un objet i18n est attendu.
4. Tout texte affiché à l'utilisateur (description, traits, prérequis, etc.) est un objet i18n `{ fr, en }`.
5. **Précédence** : si un id de pack custom égale un id SRD dans la même catégorie, le custom prime ; le `content-loader` émet un warning via `event-logger.ts` (`kind: 'custom-content.shadowed-srd'`, visibilité MJ).

### Forme minimale par catégorie

Les champs **requis** par catégorie sont listés ci-dessous. Les champs optionnels suivent les types internes de `src/shared/types/content.ts` (alignement strict — pas de format parallèle).

| Catégorie | Champs requis |
|---|---|
| **spells** | `id`, `source`, `name{fr,en}`, `level` (0-9), `school`, `castingTime{fr,en}`, `range{fr,en}`, `duration{fr,en}`, `components` (`{verbal: bool, somatic: bool, material: string\|null}`), `concentration` (bool), `ritual` (bool), `description{fr,en}`, `atHigherLevels{fr,en}\|null`, `classes` (string[] d'ids de classe) |
| **ancestries** | `id`, `source`, `name{fr,en}`, `description{fr,en}`, `traits` (array de `{name{fr,en}, description{fr,en}}`), `size`, `speed`. Sous-choix éventuels suivent la structure SRD figée par plan 13.7 (`dragonAncestries[]`, `tieflingLegacies[]`, etc.) — **réutilisée, pas réinventée**. |
| **subancestries** | `id`, `source`, `name{fr,en}`, `parentAncestryId`, `description{fr,en}` + champs optionnels par cas. Note : SRD 5.2.1 ne modélise pas les lignages comme sous-ascendances ; cette catégorie est conservée par compatibilité avec d'éventuels homebrew qui le font. |
| **classes** | `id`, `source`, `name{fr,en}`, `description{fr,en}`, `hitDie` (`d6\|d8\|d10\|d12`), `primaryAbilities` (string[]), `savingThrows` (string[]), `skillChoices` (`{count: number, from: string[]}`), `spellcasting` (object \| null) |
| **subclasses** | `id`, `source`, `name{fr,en}`, `parentClassId`, `description{fr,en}`, `level` (entier) |
| **feats** | `id`, `source`, `name{fr,en}`, `description{fr,en}`, `prerequisite{fr,en}\|null` |
| **items** | `id`, `source`, `name{fr,en}`, `category`, `description{fr,en}`, `weight` (number), + champs optionnels alignés (damage, properties, masteryProperty…) |
| **magicItems** | `id`, `source`, `name{fr,en}`, `category`, `description{fr,en}`, `rarity`, `attunement` (bool), + champs optionnels |
| **monsters** | `id`, `source`, `name{fr,en}`, `size`, `type`, `cr`, `abilities` (`{for, dex, con, int, sag, cha}`), `hp`, `ac`, `speed`, `description{fr,en}` |
| **invocations** | `id`, `source`, `name{fr,en}`, `description{fr,en}`, `prerequisiteWarlockLevel` (number \| null), + champs optionnels |
| **fightingStyles** | `id`, `source`, `name{fr,en}`, `description{fr,en}` |
| **backgrounds** | `id`, `source`, `name{fr,en}`, `description{fr,en}`, `skillProficiencies` (string[]), `toolProficiencies` (string[]), `languages` (string[]), `equipment` (string[]), `feature{fr,en}` (`{name, description}`) |
| **conditions** | `id`, `source`, `name{fr,en}`, `description{fr,en}` |

### Comportement attendu du validateur

- **Validation totale avant tout import** : aucun import partiel silencieux. Un pack avec une seule entité invalide est rejeté en entier.
- **Chaque erreur précise** : catégorie, id (ou index si id manquant), champ fautif, raison. Messages destinés à un humain (MJ).
- **Vérifications structurelles** :
  - `formatVersion` connue.
  - Unicité des `id` intra-catégorie.
  - Présence des champs requis (table ci-dessus).
  - Objets i18n bien formés (`fr` et `en` présents, chaînes non vides).
  - Types corrects.
  - **Cohérence des références internes** : `subclass.parentClassId` pointe vers une classe du pack OU du SRD ; si elle pointe vers ni l'un ni l'autre, erreur.
- **Implémentation** : JSON Schema versionné committé dans le dépôt (`schemas/grimwar-content-pack-v1.schema.json`) + passe programmatique pour les règles inter-entités (références croisées).
- **Documentation utilisateur** : `docs/CUSTOM-CONTENT-IMPORT.md` avec un exemple minimal valide par catégorie. Destiné au MJ.

## Notes for next plan

- **Export de pack** : noté en DEBT, livrable d'un plan futur. La symétrie est utile (un MJ exporte son pack pour le donner à un autre MJ) mais pas requise pour jouer à L1.
- **Migration users-scoped → campaign-scoped** (si γ retenu) : quand plan 14 livre les campagnes, ajouter une voie « copier ce pack dans une campagne ». Pas urgent ; user-scoped suffit pour S1.
- **Cache du contenu custom** : aujourd'hui Dexie cache uniquement les bundles SRD via `contentHash` (D7). Pour les packs custom, soit on étend le mécanisme (chaque pack a son hash), soit on accepte que le content-loader fasse un `onSnapshot` Firestore par campagne active (S2+ il y a un onSnapshot de toute façon). Décision tactique à arbitrer en exec — entrée DEBT si différé.
- **Plan 14 (Campaigns model)** : si γ est retenu, plan 14 n'a pas de tâche particulière liée au custom content. Si α/β, plan 14 doit prévoir la sous-collection `customContent` dans son schema.
- **Pack signé / vérifié** : pas nécessaire en v1 ; un MJ qui importe un pack le fait sous sa responsabilité. Si commercialisation v1+, on pourra ajouter un système de signature et de marketplace (post-launch backlog).
- **Limite Firestore (1Mo / doc)** : le splitting client garantit qu'aucune entité ne dépasse. Un pack de 1000 monstres avec descriptions longues atteint potentiellement plusieurs Mo total — le splitting par doc résout. Si performance read s'écroule (>10s pour résoudre 1000 entités à l'ouverture du wizard), entrée DEBT et optimisation en plan dédié.
