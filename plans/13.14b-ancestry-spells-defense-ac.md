## Notes for next plan

**1. Le pattern `commonSpellIds[]` est généralisable.** D18 a posé un précédent net : tout sort qu'une ascendance accorde « hors triplet de lignage » se déclare via le champ data-driven sur l'`Ancestry`, jamais par if/else dans `buildAncestrySpellIds()`. Si le SRD ajoute (ou si un futur pack custom apporte) un sort de trait, l'ajout est un patch JSON dans `scripts/data/srd-ancestries-l1.ts` + `pnpm content:build`, zéro code applicatif touché. Conserver ce pattern par défaut.

**2. `SpellList` ne peut plus mislabeler — exploiter cette garantie.** Le refactor prop `string | null` → `ReadonlyMap<string, string>` rend la classe entière des bugs « label global qui déborde sur un sort orphelin » structurellement impossible. Tout consommateur futur qui veut afficher une source par-sort passe par une `Map` (clé = `spellId`, valeur = libellé officiel du bundle). **Anti-pattern à refuser** : un futur PR qui retomberait sur une prop `string` globale au prétexte de simplicité = re-ouverture du trou D18. Surveiller en review.

**3. La CA dérivée vit dans `src/shared/lib/rules/ac.ts > computeDisplayedAc`.** Tout futur effet qui modifie la CA affichée (style Protection, sorts d'armure magique, magic items, conditions Bloqué/Étourdi, etc.) se branche **là**, jamais dans le composant `StatusStrip`. La fonction est pure, sans React, testable seule. Le rouge-avant-vert est trivial : muter la valeur retournée → les tests `ac.test.ts` rougissent. `character.ac` Firestore reste l'invariant `10 + modDEX` posé au build du wizard — ne **jamais** muter ce champ pour matérialiser un effet display-time (graverait une approximation dans le snapshot).

**4. La gate body-armor (`hasEquippedBodyArmor`) est l'API canonique pour distinguer « armure » vs « bouclier ».** Le SRD distingue ces deux pour Defense, certaines invocations Warlock, et plusieurs sorts. Le helper isolé dans `inventory-rules.ts` est désormais l'unique point de vérité. Un bouclier seul **n'est pas** une armure pour ces gates. Toute nouvelle règle conditionnée « si une armure est équipée » réutilise ce helper.

**5. D12 a la surface élargie post-D18 — penser au plan dédié.** La résolution D18 a ajouté thaumaturgie + communication-avec-les-animaux à la liste castable, dont communication-avec-les-animaux qui introduit le cas **« rituel + PB×/repos »** non encore implémenté (différent du « 1×/jour » Tieffelin L3/L5). Le futur plan castable d'ascendance doit couvrir 3 mécaniques distinctes : at-will (cantrips), once-per-rest counter (Tieffelin L3/L5 + Elfe L3/L5), PB-per-rest counter (Gnome forêts). Estimer le scope en conséquence (probablement 13.8c court ou couplé à « Long Rest + Daily Resources »).

**6. Le précédent « UAT visuel à 2 chemins distincts par plan » est viable.** Ce mini-plan a livré deux UAT visuels ciblés (commit 2 = sorts d'ascendance, commit 3 = StatusStrip CA), chacun avec son sous-dossier `uat-review/` purgé entre commits. Plutôt que de bundler les captures en une grosse galerie à la fin, séparer par périmètre + par commit garde l'UAT humain rapide et orienté. À continuer pour les plans qui touchent ≥2 surfaces visuelles indépendantes.
