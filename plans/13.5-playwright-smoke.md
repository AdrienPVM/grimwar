# Plan 13.5 — Playwright smoke e2e + purge dette e2e S1

> **Brief en attente de validation. Pas de code avant feu vert d'Adrien.**

## Goal

Câbler Playwright (déjà en `devDependencies`, le script `pnpm test:e2e` est déjà là — il manque la config et les tests) pour qu'un smoke headless attrape automatiquement la classe de bug « rien ne s'affiche / app ne se rend pas » qu'on vient de vivre. **Critère central** : si un futur plan rebrique la route `/` en placeholder, le smoke doit **casser**. Le smoke parcourt le chemin critique réel : `/` rend la LibraryScreen → CTA Créer → `/create` rend le formulaire → soumettre un perso minimal → atterrit sur `/character/:id` → la fiche rend (hero card + status strip + 5 mode tabs présents) → retour à la library via le nav shell → la liste contient le nouveau perso. En parallèle, ce plan **purge la dette e2e S1 accumulée** (plans 05, 06, 07, 08, 09, 10, 12, 12.5) : chaque ligne de la table consolidée devient soit un test, soit une requalification explicite.

### Séparation gate unitaire / gate e2e

- [ ] 6. **Décision tactique proposée** : `pnpm test:e2e` reste **séparé** de la triple gate (`typecheck && test && lint`) — les e2e sont plus lents (~30s+ pour un smoke complet vs <5s pour les unit tests) et nécessitent un dev server. Recommandation :
    - **Triple gate à chaque commit** = `typecheck && test && lint` (rapide, ~10s, ne bloque pas le flow)
    - **Gate e2e** = `typecheck && test && lint && test:e2e` exécutée :

        - automatiquement en CI (à ajouter dans une étape ultérieure si CI existe ; si pas de CI configurée, on documente que le gate e2e doit tourner manuellement à la fin de chaque plan S1+)
    - **DoD de ce plan et des plans suivants** : ajouter une ligne explicite « `pnpm test:e2e` vert » à la DoD des plans qui touchent UI. Mettre à jour `CLAUDE.md` section commit checks pour refléter cette double-gate.
- [ ] 7. Documenter le découpage dans le commit + dans une nouvelle note `plans/DEBT.md` section conventions ou directement dans `CLAUDE.md`. ### Smoke e2e — chemin critique

- [ ] 8. Créer `tests/e2e/smoke.spec.ts` — le **smoke central**. Un seul test, un seul parcours :
    1. `await page.goto('/')`
    2. assert : `/` rend la LibraryScreen, **pas un placeholder** — chercher `getByRole('heading', { name: /Bibliothèque/i })` ET `getByRole('button', { name: /Créer un personnage/i })`. Si l'un des deux manque → le test casse. **C'est le filet contre « rien ne s'affiche ».**
    3. assert : nav shell présent — `getByRole('navigation')` ou un `data-testid="nav-shell"`
    4. assert : `/` en empty state si le user de test est neuf (sinon, ignorer cette assert si seedé)
    5. tap CTA « Créer un personnage » → URL passe à `/create`, formulaire wizard rend (`getByRole('heading', { name: /Créer un personnage/i })`)
    6. remplir minimum requis via `createMinimalCharacter(page, { name: 'Test Hero' })`, soumettre
    7. assert : URL devient `/character/:id` (regex), hero card visible avec le nom « Test Hero », status strip visible (4 stats : HP/CA/Init/Vit), 5 mode tabs présents (Combat/Essence/Magie/Avoir/Âme)
    8. tap bouton retour du nav shell → URL repasse à `/`
    9. assert : la card du nouveau perso « Test Hero » apparaît dans la liste
- [ ] 9. **Critère de réussite du plan** explicitement testé : commenter temporairement le contenu de `LibraryScreen` (faire rendre un seul `<HeroEmblem />` comme avant 13.6) → relancer `pnpm test:e2e` → le smoke **doit échouer**. Décommenter, retest → vert. Ce sanity-check ne reste pas dans le code, juste validé une fois et noté dans le commit pour preuve. *(Ce step n'est pas un test gardé — c'est une vérification manuelle one-shot du filet.)*

### Purge de la dette e2e S1

- [ ] 10. `tests/e2e/wizard.spec.ts` — couvre **plan 05 step 24** : « Wizard manuel — un perso minimal créé en < 2min ». Test : remplir wizard avec valeurs minimales valides (point-buy ou standard array), soumettre, atterrir sur fiche. Le « < 2min » devient un assert de durée du test (`expect(durationMs).toBeLessThan(120_000)`).
    - **Sous-test « modale d'aide visible dans le viewport sans scroll »** (UAT post-plan 05 bug n°2 — anti-régression portal+overlay) : sur viewport mobile (Pixel 7 par défaut), à l'étape Classe, tap le `?` d'une carte → assert :
        - `await expect(page.getByRole('dialog')).toBeVisible()`

        - tap croix → dialog démonté ET `body overflow` restauré
